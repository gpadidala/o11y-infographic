THE OVER-ENGINEERING TEST
1
Does your O11y task always follow the same sequence of API calls?
You built a runbook script.
2
Does it always call the same Grafana / Mimir / Loki API?
You built an API wrapper.
3
Does it need to decide which backend (Mimir vs Loki vs Tempo) to query?
Now you might need an agent.
4
Can an SRE define the exact investigation steps in advance?
Workflow, not agent.
5
Does the investigation path change based on intermediate results?
Agent territory.
6
Could a wrong agent action cause a production incident?
Human-in-the-loop required.
7
Does the task require correlating 3+ signal types (metrics + logs + traces)?
Agent adds real value here.

O11Y AGENT ANTI-PATTERNS

🚫 The Dashboard Reloader

Using an agent to refresh Grafana dashboards on a schedule. This is a cron job, not an agent.

Fix: Use Grafana's built-in auto-refresh or a simple script.

🚫 The PromQL Parrot

Agent that translates the same 5 natural language queries to PromQL every time. Same input → same output = deterministic.

Fix: Build a lookup table or parameterized template.

🚫 The Alert Forwarder

Agent receives Alertmanager webhook and posts to Slack with the same format every time. This is a webhook, not intelligence.

Fix: Use Alertmanager's native Slack receiver.

🚫 The Config Copier

Agent copies datasource configs from staging to prod. Same source, same target, same transform = a script.

Fix: Use Terraform or Grafana provisioning YAML.
